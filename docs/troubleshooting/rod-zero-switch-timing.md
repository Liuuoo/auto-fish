# AF-005：鱼竿耐久归零后立即切换下一根失败

状态：已解决

## 问题描述

本地计数归零后，脚本读取到下一根可用鱼竿并立即发送数字快捷键。但游戏画面里实际没有完成切换，下一步直接抛竿失败或仍停留在旧工具状态。

## 现象

典型日志：

```text
[鱼竿] 计数耐久 4:0
[鱼竿] 快捷栏耐久 5:6/15
[鱼竿] 已发送快捷栏 5 切换按键（耐久计数归零校准）
[鱼竿] 记录当前快捷栏 5，预计剩余耐久 6
[动作] 抛竿
```

日志显示切换按键已发送，但游戏没有及时完成切换。该问题不是单个快捷栏位的问题，2、3、4、5 都可能发生。

## 根因

- 鱼竿耐久归零后，游戏需要时间完成工具用尽、装备状态刷新和输入状态恢复。
- `/game/user-item` 查询完成并不代表游戏前端已经准备好处理新的快捷栏输入。
- 如果立即发送数字键，页面可以收到事件，但游戏输入层可能忽略这次切换。

## 当前解决方案

- `_switch_to_rod_slot()` 在发送数字键前先等待 `ROD_SWITCH_PRE_KEY_DELAY = (0.16, 0.24)`，使用高斯分布，中心约 0.2 秒。
- 数字快捷键发送两次：`ROD_SWITCH_KEY_PRESSES = 2`。
- 两次数字键之间等待 `ROD_SWITCH_KEY_INTERVAL = (0.08, 0.12)`，使用高斯分布，中心约 0.1 秒。
- 切换前仍先调用 `cdp_focus_game_canvas()`，确保焦点回到游戏 `canvas`。
- 数字键仍使用 printable key 事件，避免游戏输入层忽略简化 raw 数字键事件。

## 涉及代码

- `ROD_SWITCH_PRE_KEY_DELAY`
- `ROD_SWITCH_KEY_PRESSES`
- `ROD_SWITCH_KEY_INTERVAL`
- `_switch_to_rod_slot(slot, reason="", force=False)`

## 后续维护边界

- 不要把耐久归零后的切换前等待删掉。
- 不要把数字键发送次数改回 1，除非有更可靠的游戏状态确认机制。
- 如果未来引入视觉确认或游戏内部装备状态读取，可以用真实确认替代当前时序规避。
