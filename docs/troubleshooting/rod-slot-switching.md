# AF-001：每轮点击数字快捷键导致鱼竿状态不稳定

状态：已解决

## 问题描述

旧流程每轮抛竿前都会读取 `/game/user-item`，并可能点击数字快捷键重新选择鱼竿。日志会出现“已切换到快捷栏 3/4/5”，但游戏画面里实际没有切换，随后脚本继续发送 `F` 抛竿。

这个问题不只发生在单个快捷栏，2、3、4、5 都可能出现。核心问题是脚本把“CDP 按键发送成功”误当成“游戏已完成快捷栏切换”。

## 现象

- 日志显示已发送或已切换到某个快捷栏。
- 游戏画面仍停留在旧工具或旧鱼竿状态。
- 下一步直接抛竿，导致抛竿失败、等待超时或状态偏移。

## 根因

- CDP 只能确认 `Input.dispatchKeyEvent` 调用成功，不能确认游戏逻辑已经处理快捷栏切换。
- 旧的数字键事件使用简化的 `rawKeyDown -> keyUp`，页面能收到事件，但游戏输入层不一定按快捷栏切换处理。
- 每轮都点击数字快捷键会增加触发失败的机会，把低概率输入问题放大成高频状态问题。

## 当前解决方案

- 主循环不再每轮读取接口，也不再每轮点击数字快捷键。
- 只有启动初始化、换杆、等待超时校准、预计耐久归零校准这些关键节点才允许切换快捷栏。
- `_switch_to_rod_slot()` 切换前先调用 `cdp_focus_game_canvas()`，把焦点放回游戏 `canvas`。
- 数字快捷键使用 printable key 事件发送：`keyDown` 带 `text/unmodifiedText`，再发送 `keyUp`，让页面产生正常的 `keydown -> keypress -> keyup`。
- 日志改为“已发送快捷栏 X 切换按键”，不再写“已切换”，避免把按键发送成功误解为游戏确认成功。

## 涉及代码

- `cdp_focus_game_canvas()`
- `cdp_send_key(key_def, printable=True)`
- `_switch_to_rod_slot(slot, reason="", force=False)`
- 主循环中只在本地状态缺失或校准节点调用 `ensure_usable_rod()`

## 后续维护边界

- 不要恢复为“每轮抛竿前都读取接口并点击数字快捷键”的流程。
- 不要把 CDP 返回成功当作游戏切换成功。
- 如果需要进一步确认游戏是否真的切换，应增加视觉识别、游戏内部状态读取或快捷栏坐标点击等更强确认手段。
