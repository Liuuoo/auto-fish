# AF-002：脚本收竿计数和游戏真实耐久可能不一致

状态：已解决

## 问题描述

脚本完成一次“检测咬钩 -> 收竿 -> 等待结算”的流程，不一定代表游戏真正钓上鱼，也不一定代表鱼竿耐久被消耗。旧方案如果每轮都读取接口，会让流程变慢并增加切换状态风险；如果完全不读取，又可能让本地耐久计数偏离真实耐久。

## 现象

- 脚本认为已经收竿并消耗了一次耐久。
- 游戏没有实际钓上鱼时，真实耐久不会下降。
- 长时间运行后，本地预计耐久和 `/game/user-item` 返回的真实耐久可能不一致。

## 根因

- “脚本执行了收竿动作”和“游戏实际消耗鱼竿耐久”不是等价事件。
- 游戏结算存在失败、漏钓、延迟等情况。
- 高频接口读取和高频快捷键切换会引入新的不稳定因素。

## 当前解决方案

采用“本地计数 + 关键节点接口校准”的状态机：

1. 启动或换杆时读取一次 `/game/user-item`，记录当前快捷栏和真实耐久。
2. 正常钓鱼循环不再每轮读接口。
3. 每次完成咬钩后的收竿等待流程后，本地 `ROD_EXPECTED_DURABILITY` 减 1。
4. 当预计耐久归零时，强制读取 `/game/user-item` 校准。
5. 如果当前鱼竿真实耐久为 0，则切换到下一根可用鱼竿。
6. 如果真实耐久仍大于 0，说明之前有若干次收竿没有实际消耗耐久；脚本更新 `ROD_EXPECTED_DURABILITY` 并继续使用当前鱼竿。

## 涉及代码

- `ROD_EXPECTED_DURABILITY`
- `ensure_usable_rod(reason="", force_switch=False)`
- `record_successful_reel()`
- `_ordered_available_rod_slots(slots, start_after=None)`

## 后续维护边界

- 不要重新改成每轮接口读取。
- 不要把“收竿动作已执行”直接等同于“耐久一定消耗”。
- 如果调整等待时间或收竿判定，需要保留预计耐久归零后的真实接口校准。
