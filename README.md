# auto-fish

浏览器游戏自动钓鱼辅助脚本。项目通过 Windows 窗口截图识别鱼漂和鱼影，并通过浏览器 DevTools Protocol 向已打开的游戏页面注入按键操作。

## 环境要求

- Windows 10/11
- Microsoft Edge 或 Chrome
- Python 3.12 或 3.13（仅源码运行或自行打包时需要）
- Git（仅克隆源码时需要）

源码运行所需 Python 依赖见 `requirements.txt`：

```bash
pip install -r requirements.txt
```

## 下载项目

```bash
git clone https://github.com/Liuuoo/auto-fish.git
cd auto-fish
```

## 运行方式

### 方案 1：使用已打包程序

这是推荐的日常使用方式。

1. 右键 `start_edge_debug.ps1`，选择“使用 PowerShell 运行”。
2. 在自动打开的浏览器窗口中进入目标游戏页面并登录。
3. 双击已打包的可执行文件，打开钓鱼识别程序。
4. 在程序提示下按 `F8` 绑定当前浏览器窗口。
5. 选择对应的浏览器 tab。
6. 在预览窗口中拖拽选择识别区域。
7. 按 `F7` 开始或暂停自动钓鱼。
8. 按 `Esc` 停止或退出。

如果系统禁止直接运行 PowerShell 脚本，可以改用 `StartEdge.bat` 启动浏览器调试窗口。

### 方案 2：从源码运行

1. 安装依赖：

```bash
pip install -r requirements.txt
```

2. 启动带调试端口的浏览器：

```powershell
.\start_edge_debug.ps1
```

3. 在新打开的浏览器窗口中进入目标游戏页面并登录。

4. 启动识别脚本：

```bash
python fishing0.5.py
```

5. 按程序提示操作：

- `F8`：绑定当前浏览器窗口
- 选择对应的浏览器 tab
- 在预览窗口中拖拽选择识别区域
- `F7`：开始或暂停自动钓鱼
- `Esc`：停止或退出

## 打包可执行文件

项目包含 PyInstaller 配置文件，可按需打包：

```bash
pyinstaller fishing0.5.spec
```

生成的 `build/`、`dist/`、`build_debug/` 等目录是构建产物，不纳入 Git 仓库。

## 当前用到的接口

当前脚本实际使用的外部 API/协议如下：

- Chrome/Edge DevTools Protocol：`http://127.0.0.1:9222/json`，用于列出可连接的浏览器页面，并通过 `webSocketDebuggerUrl` 连接目标 tab。
- 目标游戏前端页面：用于识别当前游戏页面来源与浏览器存储上下文。
- 目标游戏物品接口：用于读取快捷栏工具和鱼竿耐久信息。

项目不会上传本地抓包资料、接口参考文档和构建产物。

## 维护记录

### 2026-04-24：鱼竿状态与多钱包处理

#### 问题 1：每轮点击数字快捷键导致状态不稳定

现象：

- 日志中出现切换到快捷栏 4/5 后，下一轮直接发送 `F` 抛竿，但游戏内实际没有切到对应鱼竿。
- 等待咬钩多次超时，表现为脚本认为当前手持鱼竿可用，但游戏侧可能仍停留在旧状态或未成功装备。
- 旧逻辑在每轮抛竿前都会读取 `/game/user-item` 并可能点击数字快捷键，导致快捷键调用频率过高。

根因：

- CDP 只能确认按键事件已发送，不能确认游戏实际完成快捷栏切换。
- 数字快捷键在游戏内偶发被吞或未及时生效时，旧逻辑仍会把 `CURRENT_ROD_SLOT` 当作已经切换成功。
- 每轮抛竿前都重新确认和点击快捷键，会把“低概率切换失败”放大成高频风险。

当前解决方案：

- 主循环不再每轮调用鱼竿检查，也不再每轮点击数字快捷键。
- 启动、换杆、等待超时、预计耐久归零这几个节点才允许读取 `/game/user-item`。
- `_switch_to_rod_slot()` 会先把页面焦点放回游戏 `canvas`，再用带 `text/unmodifiedText` 的 printable 数字键事件发送快捷栏按键，避免游戏输入层忽略简化的 raw 数字键事件。
- 抛竿前只检查本地状态：`CURRENT_ROD_SLOT` 必须存在，`ROD_EXPECTED_DURABILITY` 必须大于 0。

#### 问题 2：真实耐久和脚本计数可能不一致

现象：

- 脚本完成了一次“检测咬钩 -> 收竿”流程，但游戏不一定真正钓上鱼。
- 没钓上鱼时，游戏真实耐久不会下降；如果每次都立即读取接口，会增加请求和状态切换负担。

根因：

- “脚本执行了收竿动作”和“游戏实际消耗鱼竿耐久”不是完全等价事件。
- 只靠每轮实时接口读取过重，只靠本地计数又可能在漏钓时偏离真实耐久。

当前解决方案：

- 启动或换杆时读取真实耐久，例如 `15/15`，记录为 `ROD_EXPECTED_DURABILITY = 15`。
- 每次完成咬钩后的收竿等待流程后，只在本地把预计耐久减 1。
- 预计耐久减到 0 时，强制读取 `/game/user-item` 做真实校准。
- 如果真实耐久是 0，认为当前鱼竿用完，按快捷栏顺序切到下一根可用鱼竿。
- 如果真实耐久仍为 1、2、3 等正数，说明之前有若干次收竿没有实际消耗耐久；此时不换杆，只把 `ROD_EXPECTED_DURABILITY` 更新为真实值后继续。

#### 问题 3：多钱包时读到错误账号

现象：

- 浏览器中存在多个钱包 session 时，接口读取可能只显示 `5:8/10`，但实际游戏装备栏不是这个状态。
- 曾经尝试按“页面最近请求地址”或“快捷栏鱼竿信息最完整”自动选择账号，但这两种规则都不能稳定代表当前游戏账号。

根因：

- 浏览器 localStorage 中可能同时保留多个 `address_session`。
- 页面最近请求的地址、接口返回物品数量、快捷栏完整程度都只是间接信号，不能作为当前账号的可靠依据。

当前解决方案：

- 多钱包时不再自动猜账号。
- 程序检测到多个 session 后，会列出编号和地址后 5 位，例如 `...abcde`。
- 用户输入编号后，本次运行固定使用该钱包请求 `/game/user-item`。
- 重新按 `F7` 启动新一轮自动钓鱼时会清空已选钱包，让用户重新选择，避免账号切换后沿用旧状态。

#### 问题 4：刚抛竿瞬间误判咬钩

现象：

- 刚发送 `F` 抛竿后，脚本可能立即检测到“咬钩”，然后立刻收竿。

根因：

- 抛竿动作刚发生时，鱼漂、鱼影、画面残留或检测框变化可能短暂满足咬钩条件。

当前解决方案：

- 抛竿后固定 `CAST_BITE_DETECTION_DELAY = 2.0` 秒不进行咬钩判定。
- 这 2 秒内仍刷新预览窗口，只跳过 `fish_touching_float()` 判定。
- 该延迟只作用于抛竿后的检测阶段，不改变收竿动作。

#### 问题 5：耐久归零后立即切换鱼竿失败

现象：

- 本地计数归零后，脚本读取到下一根鱼竿，例如日志显示 `快捷栏耐久 5:6/15`。
- 随后日志显示已发送快捷栏 5 的切换按键，并记录当前快捷栏 5。
- 游戏画面里实际仍没有完成切换，下一步直接抛竿会失败或继续停留在旧状态。

根因：

- 当前鱼竿耐久归零后，游戏需要一点时间完成工具用尽、装备状态刷新和输入状态恢复。
- 如果 `/game/user-item` 查询完成后立刻发送新数字键，页面可以收到按键事件，但游戏输入层可能尚未准备好处理快捷栏切换。
- 该问题不是单个快捷栏位的问题，2/3/4/5 都可能发生，本质是归零换杆路径的时序问题。

当前解决方案：

- `_switch_to_rod_slot()` 在发送数字快捷键前先等待 `ROD_SWITCH_PRE_KEY_DELAY = (0.16, 0.24)`，使用高斯分布，中心约 `0.2s`。
- 数字快捷键发送两次：`ROD_SWITCH_KEY_PRESSES = 2`。
- 两次数字键之间等待 `ROD_SWITCH_KEY_INTERVAL = (0.08, 0.12)`，使用高斯分布，中心约 `0.1s`。
- 仍保留切换前聚焦游戏 `canvas` 和 printable 数字键事件，避免焦点或输入事件类型导致游戏忽略切换。

#### 当前状态机

启动或重新开始：

- 清空 `CURRENT_ROD_SLOT`、`LAST_ROD_SLOTS`、`ROD_EXPECTED_DURABILITY` 和已选择钱包。
- 第一次进入运行循环时读取 `/game/user-item`。
- 多钱包时要求用户选择钱包。
- 从快捷栏 1-5 中选择第一根耐久大于 0 的鱼竿，点击对应数字键并记录真实耐久。

正常循环：

- 如果本地记录有当前鱼竿且预计耐久大于 0，直接抛竿。
- 抛竿后 2 秒内只刷新预览，不做咬钩判定。
- 检测到咬钩后等待短暂反应时间，发送 `F` 收竿。
- 收竿后等待 6-7 秒，再把本地预计耐久减 1。

耐久归零：

- 当本地预计耐久减到 0，读取 `/game/user-item`。
- 当前鱼竿真实耐久为 0 时，切换到下一根可用鱼竿。
- 当前鱼竿真实耐久大于 0 时，更新本地预计耐久并继续使用当前鱼竿。
- 切换下一根可用鱼竿前会先等待约 0.2 秒，再发送两次数字快捷键，两次间隔约 0.1 秒。

等待超时：

- 如果等待咬钩超时，读取 `/game/user-item`。
- 当前鱼竿仍可用时，强制重新点击当前快捷栏数字键做校准。
- 当前鱼竿不可用时，切换到下一根可用鱼竿。

#### 后续维护边界

- 不要恢复为“每轮抛竿前都读取接口并点击数字快捷键”的流程。
- 不要用“鱼竿数量最多”“快捷栏最完整”“最近请求地址”自动选择钱包。
- 不要把 `F` 的抛竿和收竿混成同一个状态判断；抛竿后的 2 秒检测冷却只用于抛竿阶段。
- 修改耐久逻辑时，要保留“本地计数 + 关键节点接口校准”的设计，不要让接口读取重新进入每轮循环。
- 修改换杆逻辑时，要记住 CDP 只能确认按键发送成功，不能确认游戏切换成功；耐久归零后的 0.2 秒前置等待、两次数字键发送和 0.1 秒间隔是当前规避时序问题的关键。
